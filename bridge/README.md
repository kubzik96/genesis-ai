# Bridge

## Назначение

Bridge — система передачи контекста и управления очередью работ между ИИ-агентами Genesis AI.

Её задача:
- упростить взаимодействие между агентами;
- предотвратить потерю контекста при смене исполнителя;
- обеспечить прозрачность статуса всех работ;
- фиксировать причины задержек и проблем.

---

## Принципы

- **Контекст всегда явный** — агент не должен угадывать, что нужно делать;
- **Передача перед началом** — рабочий контекст передаётся до того, как новый агент начнёт работу;
- **Проверка перед завершением** — результат проверяется перед передачей следующему агенту;
- **История видна** — каждый агент видит, что было сделано до него;
- **Единый источник истины** — все сведения хранятся в GitHub и MEMORY.md.

---

## Структура Bridge

- **QUEUE.md** — очередь активных и ожидающих работ;
- **HANDOFF.md** — формат передачи контекста между агентами;
- **MEMORY.md** (в корне проекта) — долгосрочная память между сессиями.

---

## Жизненный цикл задачи

1. **BACKLOG** — задача сформулирована, ожидает уточнения;
2. **READY** — задача готова к выполнению, есть полный контекст;
3. **WORKING** — исполнитель работает;
4. **REVIEW** — результат ожидает проверки;
5. **DONE** — задача завершена и принята;
6. **BLOCKED** — задача заблокирована зависимостью или проблемой.

---

## Основной процесс

### Для CEO / CTO

- Проверить QUEUE.md перед началом сессии.
- Убедиться, что задачи в статусе READY имеют полный контекст.
- Утвердить переход в DONE.

### Для Developer / GitHub Engineer

- Прочитать HANDOFF.md перед началом работы.
- Сообщить обо всех возникших вопросах перед началом.
- Выполнить только поставленную задачу.
- Перевести задачу в REVIEW при завершении.

### Для Reviewer (CTO / CEO)

- Проверить результат по критериям готовности.
- Утвердить или вернуть в WORKING с указанием проблем.
- Принять решение о переводе в DONE.
