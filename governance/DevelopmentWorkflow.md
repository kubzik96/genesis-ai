## 1. Цель

Определить обязательный стандарт инженерной работы Genesis AI:

- жизненный цикл от идеи до закрытия задачи;
- роли и границы ответственности;
- правила review, Execution Authorization и merge;
- разделение артефактов (DR, Specification, Bridge);
- действия при ограничениях инструментов и зависимостях между задачами.

Документ снижает зависимость процесса от конкретного ИИ-инструмента и исключает review по неподтверждённым сообщениям чата.

---

## 2. Основной принцип

**GitHub является System of Record и единственным источником истины для состояния реализации.**

Чат используется только для обсуждения, постановки задач, уточнения требований, координации и передачи решений CEO.

Сообщение агента о выполненной работе **не является** доказательством выполнения.

## 4. Разделение артефактов

| Артефакт | Вопрос | Где |
|---|---|---|
| Decision Record | *Почему?* | `decisions/` |
| Specification | *Что делать?* | `specifications/` |
| Task / QUEUE | *Кто, когда, статус?* | `bridge/QUEUE.md` |
| HANDOFF | *Какой операционный контекст?* | `bridge/HANDOFF.md` |

**Approved Specification является единственным источником функциональных требований для реализации.**

Нельзя реализовывать «по памяти», «по сообщению в чате» или «по старой версии обсуждения».

---

## 5. Роли и границы ответственности

| Роль | Ответственность |
|---|---|
| **CEO** | Стратегия, утверждение scope и спецификаций, Execution Authorization, разрешение commit/PR/merge, финальная ответственность |
| **AI-CTO / COO** | Процесс, спецификации, оркестрация, контроль Bridge, процессный review |
| **Chief Architect** | Архитектурный review, риски, независимая техническая оценка, Dissent / Stop the Line |
| **Автор спецификации** | Подготовка Draft Specification, учёт замечаний review |
| **Исполнитель (Lead / Execution Engineer)** | Реализация только по Approved Specification и Execution Authorization |
| **Независимый reviewer** | Review чужой работы; не утверждает собственную реализацию |
| **GitHub Engineer** | Техническая работа с репозиторием по утверждённым задачам; не принимает стратегические решения |
| **Другие ИИ-агенты** | В рамках назначенной роли и scope; без расширения полномочий |

**Правила:**

- Исполнитель **не может** сам утверждать собственную работу.
- Автор реализации **не должен** быть единственным reviewer.
- Один агент не выполняет изменение и окончательный независимый review своего же результата.

## 6. Жизненный цикл

```text
Idea
  ↓
Draft Specification
  ↓
Independent Specification Review
  ↓
CEO Approval
  ↓
Approved Specification (published in Git)
  ↓
Execution Authorization
  ↓
Implementation (feature branch)
  ↓
Pull Request (Ready for review ≠ merge permission)
  ↓
Independent Implementation Review
  ↓
CEO Merge Authorization
  ↓
Merge
  ↓
Post-merge Verification
  ↓
Closure (Bridge → DONE)
```

Переход на следующий этап — только после завершения предыдущего.

---

## 7. Specification

Значимая реализация начинается **только** после появления **Approved Specification** в `specifications/`.

Спецификация обязана содержать: цель, контекст, scope, ограничения, разрешённые/запрещённые файлы, критерии готовности, способы проверки, выходные артефакты, Dependencies, Assumptions, необходимость Decision Record.

### Статусы Specification

`Draft` → `In Review` → `Approved` → (`Superseded` | `Archived`)

### Revision

| Тип | Действие |
|---|---|
| **Minor** (без изменения Scope) | Новая Revision той же Specification |
| **Major** (изменение Scope / целей / критериев) | Новая Specification; предыдущая → `Superseded` |

Изменение scope после Approval → остановка + новая Revision/Specification + повторное Approval.

---

## 8. Execution Authorization

Выдаётся CEO **отдельно** от Approval спецификации.

Действует только: для одной задачи, одной Approved Revision, указанного scope, назначенного исполнителя.

**Не является** разрешением на: commit, push, PR, merge, изменение архитектуры, расширение scope, нарушение Governance.

Автоматически прекращается при конфликте с Constitution / DR / Bridge, ошибке в спецификации, необходимости архитектурного изменения, расширении scope, указании CEO остановиться.

---

## 9. Implementation и scope PR

- Работа только в feature-ветке.
- Каждый PR имеет **ограниченный и проверяемый scope**.
- Несвязанные изменения **запрещено** смешивать в одном PR.
- Зависимости между задачами выполняются в архитектурно правильном порядке (пример: T-007 до merge T-008, если T-008 ссылается на Workflow).

---

## 10. Pull Request и review

Статус **Ready for review не означает** разрешение на merge.

Merge **запрещён** без отдельного **CEO Merge Authorization**.

### Объекты review (только Git-артефакты)

- git diff / unified diff / commit / patch / PR  
- содержимое файлов из репозитория  
- результаты тестов / CI  

**Не считаются доказательством:** сообщение «готово», пересказ, предпросмотр чата, скриншот без Git-артефакта.

При расхождении между сообщением агента и Git **источник истины — Git**.

### Устаревание review

Если после review изменился HEAD PR, предыдущий verdict **устаревает**. Требуется повторная проверка актуального HEAD.

### Вердикты review

`APPROVED` | `APPROVED WITH NOTES` | `CHANGES REQUIRED` | `BLOCKED` | `HOLD`

---

## 11. Pre-merge checklist

Перед merge проверяются:

1. Актуальный HEAD SHA  
2. Scope  
3. Changed files  
4. Конфликты  
5. CI (см. §12)  
6. Reviews  
7. Зависимости между задачами  
8. Соответствие Approved Specification  
9. Отсутствие случайных изменений  

---

## 12. CI

Если CI не настроен, статус обозначается явно:

**`CI_NOT_CONFIGURED`**

Отсутствие CI **нельзя** трактовать как успешное прохождение проверок. Это ограничение ручной проверки, не «зелёный» результат.

---

## 13. Merge, verification, closure

1. Merge — только после CEO Merge Authorization.  
2. После merge — **post-merge verification** в `main` (файлы на месте, ссылки корректны, нет регрессии scope).  
3. Задача закрывается (**DONE**) только после verification.  
4. Только CEO переводит задачу в DONE.  
5. Bridge обновляется, чтобы отражать фактическое состояние.

---

## 14. Bridge

Операционные статусы фиксируются в Bridge:

`BACKLOG` → `READY` → `WORKING` → `REVIEW` → `DONE`  
(+ `BLOCKED` при необходимости)

HANDOFF — операционный контекст передачи между агентами, не замена Specification.

---

## 15. Decision Records

Обязательны, если изменение:

- вводит или заменяет архитектурный стандарт;
- меняет обязательный Workflow, роли или полномочия;
- меняет модель памяти / Bridge;
- вводит новый системный компонент;
- отменяет ранее принятое решение.

---

## 16. Процесс важнее инструмента

Если инструмент не поддерживает точечное изменение, не даёт реальный Git diff, автоматически делает запрещённое действие или искажает артефакт — исполнитель обязан остановиться, сообщить об ограничении и не менять процесс самостоятельно.

**Процесс не изменяется под ограничения инструмента без решения CEO.**

---

## 17. Stop the Line

Любой агент обязан сказать `STOP`, если:

- нарушается Constitution;
- нарушается Workflow;
- реализация выходит за Scope;
- обнаружен серьёзный архитектурный или процессный риск.

После `STOP` работа приостанавливается до решения CEO. Это не veto, а право потребовать рассмотрения риска.

---

## 18. Definition of Done

Задача DONE, только если:

- scope реализован;
- критерии готовности выполнены;
- независимые review завершены;
- CEO принял результат;
- изменения merged;
- post-merge verification пройдена;
- Bridge обновлён;
- Decision Record создан/обновлён, если требуется;
- временные артефакты не попали в репозиторий.

Commit или PR сами по себе **не означают** DONE.

---

## 19. Анти-паттерны

- реализация без Approved Specification;
- merge без CEO Merge Authorization;
- review только по сообщению агента;
- commit/merge до независимого review;
- расширение scope без остановки;
- самопроверка вместо независимого review;
- смешивание несвязанных изменений в одном PR;
- merge зависимой задачи до базовой;
- трактовка «нет CI» как «CI зелёный».
