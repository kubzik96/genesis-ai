# DR-0001 — Архитектурная структура Genesis AI

## Статус

Принято

## Дата

2026-07-22

## Автор решения

CEO Genesis AI

---

# Контекст

На начальном этапе документы Genesis AI хранились в корне репозитория.

По мере роста проекта такая структура могла привести к путанице, усложнить навигацию и затруднить работу новых ИИ-участников.

Требовалось разделить документы по их назначению и создать понятную архитектуру репозитория.

---

# Цель

Создать масштабируемую структуру, в которой каждый тип информации имеет определённое место.

---

# Рассмотренные варианты

## Вариант 1. Хранить все документы в корне

Преимущества:

- простота на раннем этапе;
- быстрый доступ ко всем файлам.

Риски:

- рост беспорядка;
- сложная навигация;
- смешение стратегии, управления и рабочих материалов.

## Вариант 2. Разделить репозиторий на смысловые области

Преимущества:

- ясная структура;
- простое масштабирование;
- понятные зоны ответственности;
- удобство для ИИ и человека.

Риски:

- необходимость поддерживать актуальные пути;
- больше каталогов;
- требуется дисциплина размещения файлов.

---

# Решение

Разделить Genesis AI на следующие смысловые области:

- `governance/` — Конституция, роли, принципы и стандарты;
- `strategy/` — Vision и Roadmap;
- `decisions/` — журнал значимых решений;
- `templates/` — повторно используемые шаблоны;
- `projects/` — создаваемые продукты;
- `archive/` — неактивные и исторические материалы.

В корне репозитория оставить:

- `README.md`;
- `MEMORY.md`;
- `ACTIVE.md`.

---

# Причины

Выбранная структура:

- отделяет постоянные правила от оперативного состояния;
- упрощает поиск информации;
- позволяет добавлять новые проекты без нарушения порядка;
- делает репозиторий понятным для новых ИИ-агентов;
- поддерживает принцип единственного источника истины.

---

# Последствия

## Положительные

- структура проекта стала прозрачной;
- появилась основа для масштабирования;
- документы получили чёткие зоны хранения;
- новые агенты смогут быстрее понимать проект.

## Отрицательные

- ссылки на документы необходимо обновлять после перемещения;
- участники должны соблюдать правила размещения файлов.

## Риски

- устаревшие ссылки могут вести к несуществующим путям;
- дублирование документов может нарушить единый источник истины.

---

# План выполнения

1. Создать каталоги.
2. Переместить управляющие документы в `governance/`.
3. Переместить стратегические документы в `strategy/`.
4. Создать рабочие каталоги.
5. Обновить README и MEMORY.
6. Проверить структуру через GitHub.

---

# Проверка результата

Решение считается выполненным, если:

- все каталоги присутствуют;
- документы находятся в правильных разделах;
- README содержит актуальные пути;
- структура доступна в GitHub;
- старые дубли файлов отсутствуют.
