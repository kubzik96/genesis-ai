# DR-0004 — Repository of Approved Specifications

## Статус

Принято

## Дата

2026-07-24

## Автор решения

CEO Genesis AI

---

# Контекст

До настоящего момента часть технических заданий существовала только в переписке между агентами.

Это создавало проблемы:

- исполнитель не имел стабильного источника требований;
- review зависел от истории чата;
- невозможно было надёжно проверить, какую Revision реализовали;
- новые агенты не могли восстановить контекст только из репозитория.

Development Workflow v1 уже предусматривает появление `specifications/`, но правило ещё не было закреплено архитектурным решением.

---

# Цель

Сделать утверждённую спецификацию официальным инженерным артефактом репозитория и обязательным источником требований до выдачи Execution Authorization.

---

# Рассмотренные варианты

## Вариант 1. Хранить только в чате

Преимущества: быстро.

Риски: потеря контекста, отсутствие истории, невозможность audit.

## Вариант 2. Хранить в Bridge

Преимущества: рядом с операционным состоянием.

Риски: смешение «что происходит» и «что нужно сделать».

## Вариант 3. Отдельный каталог `specifications/`

Преимущества:

- один источник требований;
- Git хранит историю Revision;
- любой агент работает по одному документу;
- Orchestrator сможет читать спецификации автоматически;
- соответствует паттерну `decisions/` и `05_Агенты/`.

Риски:

- дополнительный обязательный артефакт;
- нужна дисциплина обновления INDEX.

---

# Решение

Создать каталог:

```text
specifications/
├── README.md
├── INDEX.md
├── _template.md
└── archive/
```

Правила:

1. Каждая значимая задача имеет спецификацию.
2. После CEO Approval спецификация публикуется в `specifications/` со статусом `Approved`.
3. **Approved Specification является единственным источником функциональных требований для реализации.**
4. Approved Specification является **необходимым основанием** для Execution Authorization.
5. **CEO Approval спецификации сам по себе не разрешает реализацию.** Для начала Implementation требуется **отдельное Execution Authorization** CEO.
6. Minor-изменения → новая Revision. Major-изменение Scope → новая Specification (предыдущая → `Superseded`).

---

# Причины

- Соответствует принципу «Git — System of Record» (Constitution §7, Development Workflow §2).
- Завершает цикл: Idea → Specification → Review → Approval → Execution Authorization → Implementation.
- Делает процесс воспроизводимым для любых агентов.
- Не ломает существующую структуру (DR-0001).

---

# Последствия

## Положительные

- единый источник требований;
- трассируемость Revision;
- упрощение review и audit;
- подготовка к Orchestrator / Agent Registry.

## Отрицательные

- дополнительный шаг перед реализацией;
- необходимость поддерживать INDEX.

## Риски

- дублирование текста между HANDOFF и Specification (роли разделены явно);
- устаревание INDEX при отсутствии дисциплины.
