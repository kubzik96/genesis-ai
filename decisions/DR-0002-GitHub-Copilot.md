# DR-0002 — Назначение GitHub Copilot на должность GitHub Engineer

## Статус

Принято

## Дата

2026-07-22

## Автор решения

CEO Genesis AI

---

# Контекст

После завершения Фазы 1 (Основание) и перехода на Фазу 2 (Архитектура Genesis AI) проект требует постоянного исполнителя технических задач.

Cursor AI, назначенный разработчиком, временно недоступен (восстановление лимита).

Текущая архитектура Genesis AI предусматривает четыре ключевые роли: CEO, COO, CTO и Developer.

Требуется назначить ИИ-агента, который будет выполнять прикладную техническую работу: создание файлов, обновление документов, управление репозиторием и быстрые текущие изменения.

GitHub Copilot, интегрированный в GitHub, имеет необходимые возможности и доступность для этой роли.

---

# Цель

Назначить GitHub Copilot официальным GitHub Engineer проекта Genesis AI для ускорения выполнения утверждённых задач и поддержки работы CEO, COO и CTO.

---

# Рассмотренные варианты

## Вариант 1. Ждать восстановления Cursor AI

Преимущества:
- минимальные изменения в архитектуре;
- Cursor уже назначен разработчиком.

Риски:
- неизвестный срок восстановления;
- замедление работы над Roadmap;
- отложение Фазы 2.

## Вариант 2. Назначить GitHub Copilot дополнительным ИИ-агентом

Преимущества:
- GitHub Copilot уже доступен;
- можно использовать в каждой сессии;
- не требует новой инфраструктуры;
- удобен для работы с репозиторием.

Риски:
- дублирование ролей с Cursor;
- необходимо определить границы ответственности;
- требуется явное регулирование в MEMORY и ACTIVE.

## Вариант 3. Заменить Developer на комбинацию GitHub Copilot и человека

Преимущества:
- расширенные возможности;
- быстрая итерация.

Риски:
- изменение архитектуры ролей;
- усложнение процессов принятия решений;
- нарушение существующей структуры.

---

# Решение

**Назначить GitHub Copilot официальной ролью GitHub Engineer с сохранением Cursor AI в роли Developer.**

GitHub Engineer — вспомогательная техническая роль, которая выполняет:
- текущую работу с кодом и документами;
- подготовку изменений в отдельных ветках;
- автоматизацию повторяющихся действий;
- быструю обработку утверждённых задач;
- контроль качества и риски.

CTO (ChatGPT) и CEO остаются стратегическими лицами, принимающими решения.

Developer (Cursor) остаётся основным исполнителем сложных архитектурных работ, когда доступен.

GitHub Engineer (GitHub Copilot) дополняет Developer и ускоряет работу на текущем спринте.

---

# Причины назначения

## 1. Доступность

GitHub Copilot интегрирован в GitHub и доступен в каждой сессии без дополнительной настройки.

Cursor AI требует восстановления лимита, что может занять неопределённое время.

## 2. Назначение роли в данный момент

Проект находится на критическом этапе:
- завершена архитектура репозитория (DR-0001);
- начат спринт Decision System v1;
- готовится испытание кандидатов на постоянного CTO;
- требуется подготовить критерии оценки и тестовые задания.

Без дополнительной технической помощи эти задачи будут выполняться медленнее.

## 3. Соответствие архитектуре

Роль GitHub Engineer не противоречит принципам Genesis AI:
- не принимает стратегические решения;
- подчиняется CEO → COO/CTO → GitHub Copilot;
- выполняет утверждённые задачи;
- фиксирует риски и ограничения;
- все изменения хранятся в GitHub (единый источник истины).

## 4. Масштабируемость

Назначение GitHub Engineer подготавливает проект к расширению команды ИИ-агентов, предусмотренному в Roadmap.

---

# Обязанности GitHub Engineer

1. **Создание и изменение файлов** — по утверждённым требованиям.
2. **Поддержка документов** — обновление информации, ссылок, таблиц.
3. **Работа с репозиторием** — создание веток, фиксация изменений, контроль diff.
4. **Быстрая обработка задач** — выполнение текущей работы на спринте.
5. **Контроль качества** — проверка результата, выявление ошибок, предложение исправлений.
6. **Отчётность** — информирование CEO/CTO о выполнении, рисках и результатах.

---

# Ограничения GitHub Engineer

1. **Не менять решения** — Decision Records может обновлять только по приказу CEO.
2. **Не менять Конституцию** — Роли, Принципы, Стандарты меняет только CEO.
3. **Не принимать архитектурные решения** — все стратегические вопросы — к CTO/CEO.
4. **Не выполнять необратимые операции** — удаление файлов, переходы на новые платформы — только с предварительным утверждением.
5. **Требует проверки** — все работы должны быть проверены перед завершением.

---

# Последствия для проекта

## Положительные

- ускорится выполнение задач спринта Decision System v1;
- сможет быть подготовлена система испытания для CTO;
- упростится работа CEO с мелкими изменениями;
- появится надёжная поддержка технических операций;
- ИИ-агент будет постоянно доступен для текущей работы.

## Отрицательные

- появится новая роль, требующая явного разделения ответственности;
- может быть путаница между GitHub Engineer и Developer (Cursor);
- требуется актуализировать документы (MEMORY, ACTIVE, Roles).

## Риски

- неправильное разделение обязанностей может привести к конфликту ролей;
- GitHub Engineer может пересечься с другими ИИ-агентами;
- требуется регулярное обновление инструкций в GitHub-Copilot.md.

---

# План выполнения

1. ✅ Создать Decision Record DR-0002.
2. Обновить `MEMORY.md` — добавить GitHub Engineer в участников.
3. Обновить `ACTIVE.md` — добавить роль в команду.
4. Обновить `governance/Roles.md` — описать GitHub Engineer (если необходимо).
5. Обновить `decisions/INDEX.md` — добавить DR-0002 в таблицу.
6. Проверить структуру в GitHub.

---

# Проверка результата

Решение считается выполненным, если:

- Decision Record создан в `decisions/DR-0002-GitHub-Copilot.md`;
- все ссылки на DR-0002 обновлены в других документах;
- MEMORY.md и ACTIVE.md содержат актуальную информацию о GitHub Engineer;
- INDEX.md содержит запись о DR-0002;
- документ доступен в GitHub;
- все структурные изменения проверены.
